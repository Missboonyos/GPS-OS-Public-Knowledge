# Problem Model



# Purpose

The Problem Model defines how GPS OS identifies, understands, and frames operational problems.

A problem should never be defined solely by its symptoms.

GPS OS distinguishes between observable symptoms, underlying causes, operational constraints, and the desired outcomes.

The objective is to establish a shared understanding of the real problem before exploring solutions.

---

# Principle

A well-defined problem is easier to solve than a poorly understood one.

GPS OS focuses on understanding why a problem exists before determining how to solve it.

---

# Components of a Problem

Every operational problem should be described using the following elements:

- Current Reality
- Desired Reality
- Gap
- Root Cause
- Constraints
- Stakeholders
- Impact

Together, these elements provide a complete representation of the problem.

---

# Problem Discovery

Problem discovery is an investigative process rather than a brainstorming exercise.

Evidence should be gathered from multiple sources, including:

- Operational observations
- Workflow analysis
- Business rules
- Historical records
- Domain experts
- Performance data

The goal is to validate the existence and significance of the problem.

---

# Problem Representation

A problem should be represented in a structured and traceable manner.

The representation should clearly distinguish:

- Facts from assumptions
- Causes from symptoms
- Constraints from preferences
- Problems from proposed solutions

This distinction enables humans and AI to reason consistently.

---

# Relationship to GPS Kernel

The Problem Model operates between Reality and Reasoning.

Correctly framing the problem improves the quality of every downstream decision.
